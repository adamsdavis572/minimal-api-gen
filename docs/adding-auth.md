# Adding Authentication & Authorization to Generated APIs

This guide explains how to add authentication and authorization to an API generated by `minimal-api-gen`. It covers the core challenge, then describes two complementary approaches — and when to choose each one.

## The Challenge: Generated Code is Immutable

The generated code creates endpoints like this (via the `AddApiEndpoints()` or `MapAllEndpoints()` extension):

```csharp
// Generated — do not modify
public static RouteGroupBuilder MapPetEndpoints(this RouteGroupBuilder group)
{
    group.MapPost("/pet", async (...) => { ... })
        .WithName("AddPet")
        .Produces<Pet>(200);

    group.MapGet("/pet/{petId}", async (...) => { ... })
        .WithName("GetPetById")
        .Produces<Pet>(200);

    // ...
    return group;
}
```

You cannot add `.RequireAuthorization("WriteAccess")` to these generated endpoints by editing the file — regenerating the API would overwrite your changes. All authorization logic must be injected **without touching generated files**.

Two patterns solve this cleanly:

| | Approach 1 | Approach 2 |
|---|---|---|
| **Mechanism** | `IEndpointFilter` registered in DI | Custom extension method wrapping generated calls |
| **Generated code changes** | None | None |
| **New endpoints require** | Update filter mapping dictionary | Update custom extension method |
| **Per-endpoint granularity** | ✅ (via endpoint name lookup) | ✅ (via route groups or per-call chaining) |
| **Best for** | Both single-project and NuGet packaging | Primarily NuGet packaging |

## Prerequisites

Both approaches require JWT Bearer authentication and authorization policies. Install the NuGet package in your implementation project:

```bash
dotnet add package Microsoft.AspNetCore.Authentication.JwtBearer --version 8.0.0
```

Then register auth services using the `IServiceConfigurator` / `IApplicationConfigurator` scanner pattern that `Program.cs` uses automatically:

```csharp
// src/YourApi/Configurators/SecurityConfigurator.cs
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;
using YourApi.Configurators;

namespace YourApi.Configurators;

public class SecurityConfigurator : IServiceConfigurator, IApplicationConfigurator
{
    // Order=10 ensures auth middleware runs after exception handling (Order=0) but before endpoints
    public int Order => 10;

    public void ConfigureServices(IServiceCollection services, IConfiguration configuration, IHostEnvironment environment)
    {
        services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
            .AddJwtBearer(options =>
            {
                options.Authority  = configuration["Auth:Authority"];
                options.Audience   = configuration["Auth:Audience"];
            });

        services.AddAuthorization(options =>
        {
            options.AddPolicy("ReadAccess",  policy => policy.RequireClaim("permission", "read"));
            options.AddPolicy("WriteAccess", policy => policy.RequireClaim("permission", "write"));
        });
    }

    public void Configure(WebApplication app, IHostEnvironment environment)
    {
        app.UseAuthentication();
        app.UseAuthorization();
    }
}
```

Because `Program.cs` scans the assembly for `IServiceConfigurator` and `IApplicationConfigurator` implementations, no other wiring is needed — just drop the file into your project.

## Approach 1: Endpoint Filter (Recommended)

### How It Works

The generated `AddApiEndpoints()` extension automatically picks up any `IEndpointFilter` instances registered in DI and applies them to the route group:

```csharp
// Generated (endpointExtensions.mustache) — do not modify
public static WebApplication AddApiEndpoints(this WebApplication app)
{
    var group = app.MapGroup("/v2");

    // ← All IEndpointFilter services from DI are applied here automatically
    foreach (var filter in app.Services.GetServices<IEndpointFilter>())
        group.AddEndpointFilter(filter);

    PetEndpoints.MapPetEndpoints(group);
    StoreEndpoints.MapStoreEndpoints(group);
    UserEndpoints.MapUserEndpoints(group);

    return app;
}
```

This means you only need to register your filter — `Program.cs` and the generated extensions do the rest.

### Step 1 — Create the Endpoint Filter

Create a filter that maps each endpoint name to the policy it requires. Use the `EndpointNameMetadata` set by the generated `.WithName(...)` calls to look up the policy at request time:

```csharp
// src/YourApi/Filters/PermissionEndpointFilter.cs
using Microsoft.AspNetCore.Authorization;

namespace YourApi.Filters;

public class PermissionEndpointFilter : IEndpointFilter
{
    private readonly IAuthorizationService _authorizationService;

    // Maps the generated endpoint name (from .WithName()) → required policy
    private static readonly Dictionary<string, string> EndpointPolicies = new()
    {
        // Write operations
        { "AddPet",    "WriteAccess" },
        { "UpdatePet", "WriteAccess" },
        { "DeletePet", "WriteAccess" },

        // Read operations
        { "GetPetById",          "ReadAccess" },
        { "FindPetsByStatus",    "ReadAccess" },
        { "FindPetsByTags",      "ReadAccess" },
    };

    public PermissionEndpointFilter(IAuthorizationService authorizationService)
        => _authorizationService = authorizationService;

    public async ValueTask<object?> InvokeAsync(
        EndpointFilterInvocationContext context,
        EndpointFilterDelegate next)
    {
        var endpointName = context.HttpContext
            .GetEndpoint()?
            .Metadata
            .GetMetadata<EndpointNameMetadata>()?
            .EndpointName;

        // No mapping → allow (e.g. health checks, Swagger)
        if (string.IsNullOrEmpty(endpointName) ||
            !EndpointPolicies.TryGetValue(endpointName, out var policyName))
        {
            return await next(context);
        }

        var result = await _authorizationService.AuthorizeAsync(
            context.HttpContext.User, resource: null, policyName);

        return result.Succeeded
            ? await next(context)
            : Results.Forbid();
    }
}
```

### Step 2 — Register the Filter in SecurityConfigurator

Add this line to your `SecurityConfigurator.ConfigureServices`:

```csharp
// Register as IEndpointFilter so AddApiEndpoints() picks it up automatically
services.AddSingleton<IEndpointFilter, PermissionEndpointFilter>();
```

### Step 3 — No Changes to Program.cs Needed

`Program.cs` already calls `app.AddApiEndpoints()` (NuGet packaging) or `app.MapAllEndpoints()` (single project), which auto-applies all registered `IEndpointFilter` instances. Authentication middleware is wired by `SecurityConfigurator` at `Order = 10`, which runs before endpoint registration.

### Adding Auth to a New Endpoint

When the OpenAPI spec gains a new operation (e.g. `SearchPets`), regenerate, then add one line to the `EndpointPolicies` dictionary:

```csharp
{ "SearchPets", "ReadAccess" },
```

No other changes are needed.

### Pros and Cons

**Pros**
- Works automatically — no changes to `Program.cs` or generated files
- Handles new endpoints by updating one dictionary entry
- Works with both single-project and NuGet packaging configurations

**Cons**
- Auth logic is driven by endpoint names — names must match exactly
- Endpoints not in the dictionary are silently allowed through (be deliberate about what you map)

## Approach 2: Custom Extension Method (NuGet Packaging)

When the API contracts are distributed as a NuGet package (generated with `useNugetPackaging=true`), a second approach is available: write a custom extension method in your **implementation project** that calls the same `MapXxxEndpoints()` methods from the contracts package and chains additional builder capabilities onto the route groups.

This approach is more explicit — auth requirements are declared statically at registration time — but it requires a manual update any time endpoints are added to or removed from a tag.

### Generated Structure (NuGet Packaging)

```
PetstoreApi.Contracts/     ← NuGet package (endpoint definitions live here)
    Extensions/
        EndpointExtensions.cs   ← AddApiEndpoints() — no auth
    Endpoints/
        PetEndpoints.cs         ← MapPetEndpoints(group)
        StoreEndpoints.cs       ← MapStoreEndpoints(group)
        UserEndpoints.cs        ← MapUserEndpoints(group)

PetstoreApi/               ← Implementation project (add your extension here)
    Filters/
        PermissionEndpointFilter.cs
    Configurators/
        SecurityConfigurator.cs
    Extensions/
        AuthorizedEndpointExtensions.cs   ← YOUR custom extension
```

### Step 1 — Create the Custom Extension

Add a new file to the implementation project. The example below uses `PetstoreApi` as the project name — substitute your own namespace throughout. It calls the same `MapXxxEndpoints()` methods from the contracts package but applies a filter (or `.RequireAuthorization`) to the route group before delegating:

```csharp
// src/PetstoreApi/Extensions/AuthorizedEndpointExtensions.cs
using Microsoft.AspNetCore.Builder;
using PetstoreApi.Endpoints;   // from PetstoreApi.Contracts NuGet package
using PetstoreApi.Filters;

namespace PetstoreApi.Extensions;

public static class AuthorizedEndpointExtensions
{
    /// <summary>
    /// Registers all API endpoints with the permission endpoint filter applied.
    /// Use instead of AddApiEndpoints() when authorization is required.
    /// </summary>
    public static WebApplication AddAuthorizedApiEndpoints(this WebApplication app)
    {
        var group = app.MapGroup("/v2")
            .AddEndpointFilter<PermissionEndpointFilter>();

        // Delegate to generated endpoint registration methods from the Contracts package.
        // ⚠️ Update this method when new tags (endpoint groups) are added to the spec.
        // After regeneration, check the Contracts package Endpoints/ folder for new MapXxxEndpoints() classes.
        PetEndpoints.MapPetEndpoints(group);
        StoreEndpoints.MapStoreEndpoints(group);
        UserEndpoints.MapUserEndpoints(group);

        return app;
    }
}
```

### Step 2 — Use the Custom Extension in Program.cs

Replace (or comment-out) the call to the generated `AddApiEndpoints()`:

```csharp
// Program.cs
using PetstoreApi.Extensions;

// ❌ Generated extension — no authorization
// app.AddApiEndpoints();

// ✅ Custom extension — authorization enforced via PermissionEndpointFilter
app.AddAuthorizedApiEndpoints();
```

### Applying Different Auth Rules per Endpoint Group

If you need different policies for different API tags (e.g. pets are public but admin endpoints require `AdminAccess`), create separate route groups:

```csharp
public static WebApplication AddAuthorizedApiEndpoints(this WebApplication app)
{
    // Public pet browsing — only read permission required
    var publicGroup = app.MapGroup("/v2")
        .RequireAuthorization("ReadAccess");
    PetEndpoints.MapPetEndpoints(publicGroup);

    // Admin store operations — require write permission
    var adminGroup = app.MapGroup("/v2")
        .RequireAuthorization("WriteAccess");
    StoreEndpoints.MapStoreEndpoints(adminGroup);

    // User management — open (authorization handled inside handlers)
    var openGroup = app.MapGroup("/v2");
    UserEndpoints.MapUserEndpoints(openGroup);

    return app;
}
```

> **Note**: `.RequireAuthorization("PolicyName")` on a route group applies that policy to every endpoint mapped into the group. This is less granular than the filter approach — all endpoints in a group share the same policy.

### Chaining Additional Capabilities

Because the route group builder supports fluent chaining, you can stack multiple capabilities:

```csharp
var group = app.MapGroup("/v2")
    .AddEndpointFilter<PermissionEndpointFilter>() // fine-grained per-endpoint auth
    .AddEndpointFilter<RequestLoggingFilter>()     // log all requests
    .RequireRateLimiting("ApiLimiter");            // rate limiting
```

### Adding Auth to a New Endpoint

When a new tag is added to the spec (e.g. `InventoryEndpoints`), regenerate and then add one line to `AddAuthorizedApiEndpoints`:

```csharp
InventoryEndpoints.MapInventoryEndpoints(group);
```

> **Warning**: If this update is missed, the new endpoints will **not be registered at all** because `AddAuthorizedApiEndpoints` controls the full registration. Consider this when choosing between approaches.

### Pros and Cons

**Pros**
- Auth requirements are explicit and visible at a glance in one file
- Full flexibility — mix-and-match filters, `.RequireAuthorization()`, rate limiting, etc. per group
- Familiar builder pattern for ASP.NET Core developers

**Cons**
- Must be updated manually when new endpoint groups (tags) are added to the spec
- Slightly more boilerplate than Approach 1
- If forgotten, new endpoint groups are silently absent from the running API

## Choosing an Approach

| Scenario | Recommended |
|---|---|
| Single-project output (`useNugetPackaging=false`) | **Approach 1** (filter) |
| NuGet packaging, want simplicity | **Approach 1** (filter — works the same) |
| NuGet packaging, want explicit per-group auth | **Approach 2** (custom extension) |
| Need per-endpoint granularity with different policies | **Approach 1** or **Approach 2** with filter |
| Want to stack multiple capabilities (rate-limit, logging, auth) on specific groups | **Approach 2** (custom extension) |

The two approaches are also **composable**: use Approach 2 to organise endpoint groups and apply coarse-grained rules (`.RequireAuthorization`), while using Approach 1's filter inside specific groups for fine-grained per-endpoint decisions.

## Testing with Auth

See [petstore-tests.md](petstore-tests.md) for the full description of the dual-mode test infrastructure (Open Mode / Secure Mode) used by the petstore example. Key points:

- **Open Mode** (`TestMode.Open`): `BypassAuthHandler` satisfies all authorization requirements automatically — use for business logic tests that don't focus on auth.
- **Secure Mode** (`TestMode.Secure`): `MockAuthHandler` reads `X-Test-UserId` and `X-Test-Permissions` headers as JWT claims — use to test that auth rules are enforced correctly.

```csharp
// Test that write endpoint is protected
[Fact]
public async Task AddPet_WithoutWritePermission_Returns403()
{
    var factory = new CustomWebApplicationFactory { Mode = TestMode.Secure };
    var client = factory.CreateClient();

    var request = new HttpRequestMessage(HttpMethod.Post, "/v2/pet")
    {
        Content = JsonContent.Create(new { name = "Fluffy", status = "available" })
    };
    // Provide read permission only — write should be denied
    request.Headers.Add("X-Test-UserId", "user1");
    request.Headers.Add("X-Test-Permissions", "read");

    var response = await client.SendAsync(request);

    response.StatusCode.Should().Be(HttpStatusCode.Forbidden);
}
```

For end-to-end integration tests using the Bruno CLI, see the `bruno/OpenAPI_Petstore/auth-suite/` folder and run:

```bash
devbox run task test:petstore-integration SUITE="all-suites-with-auth"
```
